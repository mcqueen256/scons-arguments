   1  """`SConsArguments` 
   2   
   3  This module provides an implementation of SCons *Arguments*. An *Argument* is 
   4  an entity which correlates up to three things: 
   5   
   6  - single construction variable in SCons environment (``env['NAME'], env.subst('$NAME')``), 
   7  - single SCons command-line variable (``variable=value``), and 
   8  - single SCons command-line option (``--option=value``). 
   9   
  10  Some of the above may be missing in *Argument* definition, so we may for 
  11  example correlate only a construction variable with command-line option (no 
  12  command-line variable in definition). 
  13   
  14  *Arguments* define how information flows from command-line to SCons environent. 
  15  They also may sink values from OS environment and persist them to a file. 
  16   
  17  *Arguments* use separate "namespaces" for environment variables, command-line 
  18  variables and command-line options; the user then defines mappings between 
  19  these. For example, you may create an *Argument* named ``foo`` which correlates 
  20  a construction variable named ``ENV_FOO``, command-line variable named 
  21  ``VAR_FOO`` and command-line option identified by key ``opt_foo`` (we use 
  22  ``dest`` attribute of command line option as its identifying key, see `option 
  23  attributes`_ of python ``optparse``). At certain point you request *Arguments* 
  24  to update your SCons environment ``env``, that is to populate environment with 
  25  values taken from command-line variables and/or options.  At this point, value 
  26  taken from command-line variable ``VAR_FOO`` or value from command-line option 
  27  ``opt_foo`` is passed to construction variable ``ENV_FOO``. If both, 
  28  command-line variable and command-line option are set, then command-line option 
  29  takes precedence. 
  30   
  31  If a command-line value is a string, it may contain substitutions (e.g. 
  32  ``VAR_FOO`` may be a string in form ``"bleah bleah ${VAR_BAR}"``). 
  33  Placeholders are assumed to be variable/option names in "local namespace". It 
  34  means, that if we have a command-line variable, and its value is 
  35  a string containing placeholder ``"$VVV"``, then ``VVV`` is assumed to be the 
  36  name of another command-line variable (and not, for example, construction or 
  37  environment variable). When passing strings from command-line variables and 
  38  options to an environment, the placeholders are renamed to represent 
  39  corresponding construction variables in SCons environment. This is 
  40  shown in the example below. 
  41   
  42  **Example** 
  43   
  44  Assume, we have the following three *Arguments* defined:: 
  45   
  46      .               (1)         (2)         (3) 
  47      Arguments:      foo         bar         geez 
  48      Environment:    ENV_FOO     ENV_BAR     ENV_GEEZ 
  49      Variables:      VAR_FOO     VAR_BAR     VAR_GEEZ 
  50      Options:        opt_foo     opt_bar     opt_geez 
  51      .             --opt-foo   --opt-bar   --opt-geez 
  52   
  53   
  54  and we invoked scons as follows:: 
  55   
  56      # Command line: 
  57      scons VAR_FOO='${VAR_BAR}' VAR_BAR='${foo}' --opt-geez='${opt_foo}' 
  58   
  59  then, after updating a SCons environment ``env`` with *Arguments*, the 
  60  environment will have the following construction variables set:: 
  61   
  62      env['ENV_FOO'] = '${ENV_BAR}'   # VAR_FOO -> ENV_FOO,  VAR_BAR -> ENV_BAR 
  63      env['ENV_BAR'] = '${foo}'       # VAR_BAR -> ENV_BAR,  foo -x-> foo 
  64      env['ENV_GEEZ'] = '${ENV_FOO}'  # opt_geez-> ENV_GEEZ, opt_foo -> ENV_FOO 
  65   
  66  The arrow ``-x->`` denotes the fact, that there was no command-line variable 
  67  named ``foo``, so the ``"${foo}"`` placeholder was left unaltered. 
  68   
  69  Now, as we know the general idea, let's see some code examples. 
  70   
  71  **Example** 
  72   
  73  Consider following ``SConsctruct`` file 
  74   
  75  .. python:: 
  76   
  77      from SConsArguments import ArgumentDecls 
  78      env = Environment() 
  79      decls = ArgumentDecls( 
  80         # Argument 'foo' 
  81         foo =  (   {'ENV_FOO' : 'ENV_FOO default'},                  # ENV 
  82                    ('VAR_FOO',  'VAR_FOO help'),                     # VAR 
  83                    ('--foo', {'dest' : "opt_foo"})         ),        # OPT 
  84         # Argument 'bar' 
  85         bar = (   {'ENV_BAR' : None},                                # ENV 
  86                   ('VAR_BAR', 'VAR_BAR help', 'VAR_BAR default'),    # VAR 
  87                   ('--bar',  {'dest':"opt_bar", "type":"string"})),  # OPT 
  88         # Argument 'geez' 
  89         geez =(   {'ENV_GEEZ' : None},                               # ENV 
  90                   ('VAR_GEEZ', 'VAR_GEEZ help', 'VAR_GEEZ default'), # VAR 
  91                   ('--geez', {'dest':"opt_geez", "type":"string"}))  # OPT 
  92      ) 
  93      variables = Variables() 
  94      args = decls.Commit(env, variables, True) 
  95      args.UpdateEnvironment(env, variables, True) 
  96   
  97      print "env['ENV_FOO']: %r" %  env['ENV_FOO'] 
  98      print "env['ENV_BAR']: %r" %  env['ENV_BAR'] 
  99      print "env['ENV_GEEZ']: %r" %  env['ENV_GEEZ'] 
 100   
 101  In above we define three *Arguments*: ``foo``, ``bar`` and ``geez``. 
 102  Corresponding construction variables (environment) are named ``ENV_FOO``, 
 103  ``ENV_BAR`` and ``ENV_GEEZ`` respectively. Corresponding command-line variables 
 104  are: ``VAR_FOO``, ``VAR_BAR`` and ``VAR_GEEZ``. Finally, the command-line 
 105  options that correspond to our *Arguments* are named ``opt_foo``, ``opt_bar`` 
 106  and ``opt_geez`` (note: these are actually keys identifying options within 
 107  SCons script, they may be different from the option names that user sees 
 108  on his screen - here we have key ``opt_foo`` and command-line option ``--foo``).
